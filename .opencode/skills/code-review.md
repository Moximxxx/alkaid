# Code Review Skill

## 使用场景

当用户要求进行代码审查、审查 PR、或审查代码变更时使用此 Skill。

## 审查流程

### 1. 理解变更范围

```
审查任何代码前，必须先理解：
1. 变更了什么文件
2. 变更的目的是什么
3. 是否符合项目约束
```

### 2. 安全性审查

检查以下安全问题：

| 检查项 | 说明 | 关键词 |
|--------|------|--------|
| SQL 注入 | 参数化查询 | `execute`, `query` + 字符串拼接 |
| XSS | 输出编码 | `innerHTML`, `dangerouslySetInnerHTML` |
| 敏感信息 | 密钥、日志 | `password`, `token`, `secret` |
| 权限验证 | 访问控制 | `auth`, `permission`, `role` |

```bash
# 敏感信息检查
grep -rn "password\|token\|secret\|api_key" --include="*.py" --include="*.js"
```

### 3. 性能审查

| 检查项 | 说明 | 阈值 |
|--------|------|------|
| N+1 查询 | 循环内查询数据库 | 禁止 |
| 大文件加载 | 内存占用 | > 10MB |
| 同步阻塞 | 事件循环 | 禁止 |
| 缓存未命中 | 重复计算 | 检查 |

```bash
# 检查 N+1 查询模式
grep -rn "for.*in.*:" --include="*.py" | while read line; do
    # 检查循环内是否有数据库查询
done
```

### 4. 可维护性审查

| 检查项 | 说明 |
|--------|------|
| 代码重复 | DRY 原则 |
| 函数过长 | > 50 行需重构 |
| 命名规范 | 意图明确 |
| 注释质量 | 解释为什么，不是做什么 |

### 5. 测试覆盖

| 变更类型 | 要求 |
|----------|------|
| 新功能 | 必须有测试 |
| Bug 修复 | 必须有回归测试 |
| 性能优化 | 必须有性能测试 |
| 安全变更 | 必须有安全测试 |

## 审查清单

```markdown
- [ ] 变更范围明确
- [ ] 安全检查通过
- [ ] 性能检查通过
- [ ] 代码规范符合
- [ ] 测试覆盖充分
- [ ] 文档已更新
- [ ] 没有引入新的技术债务
```

## 常见问题模式

### 高风险模式（必须修复）

```python
# SQL 注入
cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")

# 硬编码密钥
API_KEY = "sk-1234567890abcdef"

# 危险的文件操作
os.system(f"rm -rf {user_input}")

# 不安全的反序列化
pickle.loads(data)
```

### 中风险模式（建议修复）

```python
# 全局状态
global counter

# 循环内创建大对象
for item in large_list:
    df = process(item)  # 每次创建新 DataFrame

# 同步 HTTP 请求
response = requests.get(url)  # 使用 asyncio
```

## 验证命令

```bash
# 运行 linter
ruff check .

# 运行类型检查
mypy .

# 运行安全检查
bandit -r .

# 运行测试
pytest --cov=.
```
