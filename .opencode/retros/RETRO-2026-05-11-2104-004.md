## 问题分析

### 问题1：ESLint 配置缺失导致 verify 链中断
- **问题**: `bun run verify` 串联 typecheck→lint→test，因缺少 `.eslintrc.json` 导致 lint 报错、verify 短路失败
- **根因**: package.json 中定义了 `lint` 命令（`eslint src --ext .ts,.tsx`），但项目初始化时未同步创建 ESLint 配置文件。该问题在 FIX-009 之前未暴露，因为之前没有 `verify` 串联命令，lint 从未被调用
- **影响**: FIX-009 验证失败，驱动创建 FIX-011 专项修复

### 问题2：bun audit 服务不可用
- **问题**: `bun audit` 返回 HTTP 404（registry.npmjs.org 对该端点无响应）
- **根因**: bun 的 npm 兼容层中，`bun audit` 功能可能尚未稳定或项目依赖格式导致端点解析失败
- **影响**: FIX-010 引入 `bun audit` 后无法运行，需 FIX-012 回退为 `npm audit`

### 问题3：grep -P 在 Windows 下不可用
- **问题**: verify_arch.sh 使用 `grep -oP`（Perl 正则），Windows Git Bash 内嵌的 grep 不支持 -P 标志
- **根因**: 原始脚本为 Linux 环境编写，未考虑 Windows 兼容性
- **影响**: verify_arch.sh 第 140 行报错退出

### 问题4：自优化迭代的递归修改风险（已规避）
- **问题**: 本次自优化涉及修改 AGENTS.md、coordinator.md、contract-schema.json 等 Agent 工作流核心文件
- **根因**: 修改自身运行逻辑存在"刀削刀把"风险——优化代码本身可能存在缺陷，导致后续所有 Agent 行为异常
- **规避方案**: PLAN-004 设计分批执行，批次A（零风险基础修复）先行验证整套流水线健康，批次B/C（修改 Agent 行为）在后。每批均通过 typecheck→lint→test→build 四级门禁

## 经验教训

### 1. 分批迭代降低自修改风险
PLAN-004 将自优化拆分为 3 批，批次A（零风险）先行验证基础设施健康后再修改 Agent 核心行为（批次B/C），避免了"刀削刀把"——即用有缺陷的自优化代码进一步优化自身的递归故障。

### 2. 验证串联链的价值
FIX-009 新增的 `bun run verify`（typecheck→lint→test 串联）在批次A中立即暴露了两个隐藏问题：
- ESLint 配置缺失（lint 不可运行）
- bun audit 端点不可用（audit 不可运行）
如果没有这个串联验证链，这些问题可能在后续开发中才意外触发。

### 3. Shell 脚本跨平台兼容性是常见盲区
verify_arch.sh 的 grep -P 在 Windows Git Bash 下不可用、WSL 路径 `/mnt/d/...` 与 Windows `D:\...` 差异——这些问题表明：`.opencode/scripts/` 目录中的脚本应当假设在 Windows Git Bash 环境执行，遵循 POSIX 最小公分母（grep -E 而非 -P、避免 /proc/ 文件系统操作）。

### 4. 自动修复循环的价值与前提
批次B 实现的 R-14（自动修复循环 ≤3 次）是本次自优化的核心产出之一。回顾批次A 的手动修复链（FIX-009→FIX-010→FIX-011→FIX-012），这个过程恰好是 4 次重试（含初始执行），说明自动修复循环的设计合理。但 R-14 的实施前提是先有 ESLint 配置和完整的验证链——这也验证了批次A 作为基础设施修复必须先行。

### 5. 全链路 Trace ID 的落地成本低但收益高
FEAT-010 在所有 9 个 Agent prompt 中追加 Trace ID 字段的实际改动仅仅是每个文件的报告模板加一行 `- Trace ID：[trace_id]`，改动量小但为后续事故排查、日志关联提供了全链路可追溯性。

### 6. 技能集成的渐进式设计
批次C 采用"plan 建议 → coordinator 加载"的模式，而不是直接在 coordinator 中硬编码技能映射表。这种设计让技能成为可扩展的配置项，后续新增技能只需在 `.opencode/skills/` 添加文件并在 plan prompt 中更新参考表即可。
