## 关键架构决策（ADR）

### ADR-001: 多模态异步并行管线

**决策**: 采用分离式管道架构，Camera -> VisionPipeline + STT -> ContextMerge -> LLM -> TTS，每阶段独立可暂停/中断。

**理由**:
- 每个阶段独立封装为服务（src/renderer/services/），职责单一
- InterruptController 统一管理 AbortSignal 传播，VAD 检测到说话 -> 立即停止 TTS + 中止 LLM 流
- ContextManager 滑动窗口按 token 估算裁剪，优先丢弃最早视觉帧

**替代方案**: 全耦合单体 hook（如最初 useVideoChat）——扩展性差，FEAT-006 后重写为管线

### ADR-002: 双场景系统提示词

**决策**: text_chat（功能型助手）与 video_call（AI 伴侣）共享基座 {system_core}，通过场景差异化字段组合。

**理由**:
- 同一应用需支持两种交互模式，但不需要两套代码路径
- SystemPromptBuilder.build(scenario) 运行时组装，支持 aiName / userLanguage 可配
- AI 伴侣人格定义在 SYSTEM_PROMPTS.video_call 中，温柔/细心/主动观察

### ADR-003: VAD 基于 Web Audio API

**决策**: 使用浏览器原生 AudioContext + AnalyserNode 做 RMS 能量检测，而非引入外部 ML 模型。

**理由**:
- 零依赖，轻量级（VAD 每 100ms 计算一次 RMS）
- 可配置阈值 + 静音超时，适配不同设备
- 通过 PIPELINE_DEFAULTS.vadThreshold = 0.3 统一管理

### ADR-004: 通话状态机 useReducer

**决策**: 使用 useReducer 而非 useState 管理 idle -> calling -> connecting -> connected -> ended 状态流转。

**理由**:
- 5 个状态 + 11 种 Action，useReducer 天然适合状态机
- TICK Action 驱动秒级计时器，connected 时自动递增
- 对照组 Props（controls、aiStatus）分离管理，避免单一巨型 state

### ADR-005: 帧降分辨率 + 简单去重

**决策**: vision-pipeline 捕获帧后降分辨至 320x240，比较 base64 长度差异做去重。

**理由**:
- 降低 API token 消耗（单帧从 720p ~500KB 降至 ~3KB）
- 简单去重（frameDedupThreshold: 0.05）避免发送相同帧
- 扩展性：预留 SSIM 接口（当前为简化实现）

## 改进建议

### 1. 管道可视化调试工具
- 当前: 管线状态仅通过 aiStatus 枚举暴露为 UI 状态指示器
- 建议: 增加管线可视化面板（PipelineDebugPanel 组件），实时显示每阶段耗时、队列长度、帧去重率
- 优先级: P2（可选，调试需要时实现）

### 2. 帧分析结果缓存
- 当前: 相同帧去重仅在 vision-pipeline 内部比较相邻帧
- 建议: 增加跨请求帧哈希缓存（Map<hash, timestamp>），避免短时间内对相似场景重复发送 API
- 优先级: P2（成本优化，非关键）

### 3. TTS 语音选择器
- 当前: TTS 使用浏览器默认语音，无用户可选设置
- 建议: 在 Settings 页面增加 TTS 语音选择器（speechSynthesis.getVoices()）
- 优先级: P3（体验优化）

### 4. 端到端 WebRTC 真实视频流
- 当前: 摄像头采集为本地 MediaStream 帧截图发给 AI，无真正的 WebRTC 双向视频传输
- 建议: 若未来需要远程通话，需引入 WebRTC 信令服务器
- 优先级: P4（远景功能）

## 经验教训

1. **分批迭代降低风险**: PLAN-003 将大功能拆分为 4 个独立可验证的批次，每批次都经过 typecheck+build+test 门禁后才进入下一批，避免了大规模重构的一次性风险。

2. **管线架构的演进性**: FEAT-005 实现了 UI 层，FEAT-006 将原有单体 useVideoChat 拆分为 5 个独立服务 + VAD，这种"先 UI 后管线"的顺序让每一层都有充分的验证窗口。

3. **系统提示词的模板化设计**: 将 text_chat 和 video_call 的提示词存储在 constants.ts 而非硬编码在 service 中，让 AI 人格调整无需改动业务逻辑代码。

4. **合同驱动开发的纪律**: 5 个合同都严格定义了 coverage_checklist 的 assert 项，使得每个 PR（FEAT）的完成标准可量化、可自动化验证。
