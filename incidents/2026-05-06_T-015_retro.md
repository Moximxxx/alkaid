## 根因分析

### 技术选型：原生 fetch → LangChain

**原因**：
1. 统一不同 provider 的调用方式
2. LangChain 提供更好的流式处理抽象
3. 便于后续扩展（如添加缓存、检索等能力）

**实现方式**：
- `ChatOpenAI` 类封装了 OpenAI 兼容接口
- 通过 `configuration.baseURL` 支持自定义 endpoint
- `streaming: true` + `stream()` 实现流式响应

## 经验教训

### 1. 抽象层选择的权衡

**问题**：LangChain 增加了包体积（`langchain` + `@langchain/core` + `@langchain/openai`）

**教训**：
- 引入外部库需要权衡：开发效率 vs 包体积
- 对于简单的 API 调用，原生 fetch 可能更轻量
- 建议：如果只做简单调用，考虑只使用 `@langchain/openai`（不引入完整的 `langchain` 包）

### 2. Provider 兼容性仍需注意

**现状**：
- 所有 provider 通过 `baseURL` 配置实现 OpenAI 兼容
- 图片消息格式需要 provider 特定处理（Claude vs 其他）

**教训**：
- LangChain 提供了统一抽象，但仍需关注 provider 差异
- 建议维护 provider 特性矩阵文档

### 3. 构建验证的重要性

**验证结果**：Task-T-014 构建验证通过

**意义**：
- 依赖变更必须通过构建验证
- 确认产物完整性（`dist/`）
