## 问题分析

### 问题 1：coordinator-guard.sh 脚本缺陷（严重）

**描述**：`check_retro` 函数在 `set -e` 环境下存在逻辑问题

**根因**：
```bash
set -e  # 启用严格模式
check_retro() {
    if [ -f "$retro_flag" ]; then
        exit 1  # 直接退出，而非返回错误码
    fi
}
```
当 retro_required 文件存在时，`exit 1` 直接终止脚本，但函数没有显式返回值，`set -e` 可能导致意外行为。

**影响**：在严格模式下，脚本行为不可预测

**修复建议**：
```bash
check_retro() {
    local retro_flag="/tmp/opencode/.retro_required"
    if [ -f "$retro_flag" ]; then
        echo "BLOCKED: Previous task requires Retro review..."
        return 1  # 使用 return 而非 exit
    fi
}
```

## 经验教训

1. **脚本严格模式需谨慎**：在 `set -e` 环境下，函数内使用 `exit` 而非 `return` 会导致脚本直接终止，应统一使用 `return`

2. **验证脚本需要覆盖所有强制规则**：R-7（No-Skip-Coordinator）是核心护栏规则，必须有自动化检查

3. **Guard 脚本应有测试机制**：建议增加 dry-run 模式用于验证脚本本身的正确性
