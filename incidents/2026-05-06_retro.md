## 问题描述

AI消息流式输出时出现两个问题：
1. **loading占位符不消失**：AI首token到达后，loading动画仍在显示
2. **消息重复**：loading消息未被正确替换，导致界面出现重复消息

---

## 根因分析

| 阶段 | 问题 | 根因 |
|------|------|------|
| `fc5348e` 初始实现 | loading消息ID与AI回复消息ID不一致 | `sendMessage` 未接收 `streamingId` 参数，导致 `ai.ts` 内部生成随机ID `${Date.now()}-assistant`，而 `Home.tsx` 生成 `loading-${Date.now()}` |

**问题链路**：
```
Home.tsx: tempLoadingId = "loading-1746544xxx"
    ↓ 调用 sendMessage(contentToSend) [未传ID]
ai.ts: assistantMsg.id = "${Date.now()}-assistant" = "1746544yyy"  ← ID不一致！
    ↓ callback 返回 messages
Home.tsx: setMessages 尝试替换 loading 消息但 ID 不匹配 → 替换失败 → 消息重复
```

---

## 修复方案

### 修改文件

**1. `src/renderer/services/ai.ts`**

| 修改点 | 说明 |
|--------|------|
| `sendMessage` 签名 | 新增可选参数 `streamingId?: string` |
| `assistantMsg.id` | 优先使用传入的 `streamingId`，否则降级到 `${Date.now()}-assistant` |

```typescript
// 修改前
const sendMessage = async (content: string, image?: string): Promise<void> => {
  const assistantMsg: ChatMessage = {
    id: `${Date.now()}-assistant`,  // 随机ID
    ...
  }
}

// 修改后
const sendMessage = async (content: string, image?: string, streamingId?: string): Promise<void> => {
  const assistantMsg: ChatMessage = {
    id: streamingId || `${Date.now()}-assistant`,  // 优先用传入的ID
    ...
  }
}
```

**2. `src/renderer/pages/Home.tsx`**

| 修改点 | 说明 |
|--------|------|
| `sendMessage` 调用 | 传入 `tempLoadingId` 作为 `streamingId` |

```typescript
// 修改前
await sendMessage(contentToSend)

// 修改后
await sendMessage(contentToSend, undefined, tempLoadingId)
```

## 经验教训

### 1. 状态同步必须保证ID一致性

**问题**：UI状态（loading消息）与数据状态（AI回复）通过ID关联，ID不一致导致状态无法同步。

**教训**：
- 涉及多组件/多模块状态同步时，必须确保关联字段（如ID）的一致性
- 临时ID应在整个操作链中保持传递，而非在不同位置独立生成

### 2. 回调模式中的数据一致性

**问题**：`useAI` 内部生成的ID与调用方预期的ID不同，导致callback更新时无法正确匹配。

**教训**：
- 回调模式中，如果涉及ID引用，应提供ID参数由调用方指定
- 或者约定ID生成规则，确保双方使用相同的生成方式

### 3. 单元测试的重要性

**反思**：如果初始实现时有针对ID一致性的测试用例，此问题可在早期发现。

**建议**：对于状态管理逻辑，应添加简单的ID一致性验证测试。
