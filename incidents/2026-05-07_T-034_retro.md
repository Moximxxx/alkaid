## 问题分析

### 现象

- AI 数据被正确提取（Extracted content 日志正常）
- 但 UI 没有更新显示

### 根因分析（初步）

**可能原因 1：React 状态更新批处理问题**

在 `Home.tsx` 中，`setMessageUpdateCallback` 回调内使用 `setMessages` 更新状态：

```typescript
setMessageUpdateCallback((msgs) => {
  for (const msg of msgs) {
    setMessages((prev) => {
      // ...
    })
  }
})
```

这里对每条消息都单独调用一次 `setMessages`，可能触发 React 18 的自动批处理机制，导致状态更新被合并或丢弃。

**可能原因 2：消息 ID 不匹配**

- Home.tsx 使用 `tempLoadingId`（如 `loading-1746537600000`）创建 loading 消息
- ai.ts 使用相同的 `streamingId` 创建 assistant 消息
- 如果 ID 格式不一致，更新时会找不到现有消息

**可能原因 3：消息渲染逻辑问题**

在 `Home.tsx` 第124行添加的调试日志可以验证渲染是否被调用：

```typescript
{(() => { console.log('[Home] Rendering message:', message.id, message.role, message.content.length); return null; })()}
```

如果此日志未打印，说明消息根本没有进入渲染流程。

## 经验教训

### 1. React 状态更新应尽量使用函数式更新而非循环内多次更新

**问题**：在循环内多次调用 `setMessages` 可能导致批处理问题。

**教训**：
```typescript
// 不推荐（多次状态更新）
for (const msg of msgs) {
  setMessages((prev) => ...)
}

// 推荐（单次状态更新）
setMessages((prev) => {
  const newMessages = [...prev]
  for (const msg of msgs) {
    const index = newMessages.findIndex((m) => m.id === msg.id)
    if (index >= 0) {
      newMessages[index] = { ...newMessages[index], content: msg.content }
    } else {
      newMessages.push({ id: msg.id, role: msg.role, content: msg.content })
    }
  }
  return newMessages
})
```

### 2. 调试日志应在关键路径上覆盖完整调用链

**问题**：日志分散在多个文件，难以串联分析。

**教训**：
- ai.ts: 记录 sendMessage、Raw line、Parsed、Extracted content
- Home.tsx: 记录 messageUpdateCallback、setMessages、Rendering
- 需要确保日志能够串联完整的调用链

### 3. 未提交的调试代码应尽快清理或提交

**问题**：Home.tsx 有6行调试日志未提交。

**教训**：
- 调试完成后应立即清理 console.log
- 如果需要保留调试代码，应使用 DEBUG 标志控制
- 避免调试代码污染生产代码
