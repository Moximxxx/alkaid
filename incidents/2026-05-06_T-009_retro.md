## 重要发现

### Stream 参数与响应格式不一致

**现象**：
- 请求已正确设置 `stream: true`
- 但API返回的是**非流式完整响应**而不是SSE流式格式
- 可能是特定provider（minimax）的API兼容性问题

**初步分析**：
```
请求：{ stream: true }
预期响应：SSE 流式格式 (data: {...}\n\n)
实际响应：完整 JSON 响应 { choices: [...], ... }
```

**可能原因**：
1. minimax API 对 `stream` 参数的处理方式与标准 OpenAI 兼容
2. API 版本或端点差异
3. 请求体格式不匹配导致API降级到非流式模式

**建议**：
- 需要进一步对比 minimax API 文档确认 `stream` 参数的正确用法
- 检查请求头 Content-Type 是否正确
- 确认 API 端点是否支持流式响应

## 经验教训

### 1. API 兼容性需要明确验证

**问题**：设置 `stream: true` 但未得到流式响应，说明对特定provider的API行为了解不足。

**教训**：
- 每个 AI provider 的 API 行为可能存在差异
- 不能假设 OpenAI 兼容 API 的行为完全一致
- 需要针对每个 provider 进行单独的流式输出验证

### 2. 调试时需要保留完整的请求/响应日志

**反思**：如果当时捕获了完整的 HTTP 请求和响应，可以更快定位是请求问题还是响应处理问题。

**建议**：
- 添加请求调试日志（请求头、请求体）
- 添加响应调试日志（响应头、响应类型、首字节时间）
- 对于流式响应，记录首次数据到达时间

### 3. 流式输出的两端验证

**反思**：流式输出涉及两端：
- **发送端**：API 请求和响应接收
- **接收端**：SSE 解析和 UI 更新

**教训**：
- 发现问题时需要明确是哪个端的问题
- 可以使用 `curl` 直接测试 API 响应格式，排除客户端代码问题
