## 本次修改内容

### CORS头修复（Task-T-026）

**变更位置**：`electron/main.cjs:131-143`

**问题**：代理服务器 localhost:3000 没有设置 CORS 头，导致前端请求被浏览器拦截：
```
No 'Access-Control-Allow-Origin' header is present on the requested resource
```

**解决方案**：在 `startProxyServer` 函数中添加：
1. OPTIONS 预检请求处理（第136-142行）
2. CORS 响应头（Access-Control-Allow-Origin, Access-Control-Allow-Methods, Access-Control-Allow-Headers）

**修改后代码**：
```javascript
if (req.method === 'OPTIONS' && req.url === '/api/ai/chat') {
  res.setHeader('Access-Control-Allow-Origin', '*')
  res.setHeader('Access-Control-Allow-Methods', 'POST, OPTIONS')
  res.setHeader('Access-Control-Allow-Headers', 'Content-Type, Authorization')
  res.end()
  return
}
```

**验证**：
- ✅ `bun run build` 构建成功
- ✅ CORS头已添加到代理路由
- ✅ OPTIONS预检请求正确处理

## 根因分析

### 问题：CORS跨域限制

**现象**：
- 前端页面（http://localhost:5173）通过代理服务器（http://localhost:3000）调用AI API
- 浏览器安全策略阻止了跨域请求

**根因**：
- Electron应用的渲染进程与代理服务器不在同一源
- 代理服务器未返回正确的CORS响应头

**修复方案**：
- 在代理服务器中添加 `Access-Control-Allow-Origin: *` 头
- 处理 OPTIONS 预检请求

---

## 经验教训

### 1. 代理服务器必须处理预检请求

**问题**：OPTIONS请求未正确处理。

**教训**：
- 浏览器在发送POST请求前会先发送OPTIONS预检请求
- 预检请求必须立即返回，不能等待body数据
- 预检响应不包含body，只需要CORS头

### 2. Electron开发模式下的CORS处理

**问题**：开发环境下前端和Electron代理服务器存在跨域。

**教训**：
- 使用Electron主进程作为代理可以绕过CORS限制
- 需要在主进程中正确配置CORS头
- 生产环境同样需要考虑CORS配置

### 3. 简单请求与非简单请求的区别

**教训**：
- 简单请求（Content-Type: application/json）也会触发预检
- 必须同时处理POST和OPTIONS两种方法
