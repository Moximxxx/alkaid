**问题描述**：
verify_arch.sh 第 228-230 行使用 `grep -c 'doubao-'` 全局统计，导致将 `DEFAULT_VISION_CONFIG.model` 和 `DEFAULT_AI_CONFIG.model` 中的默认值也计入。

**根因**：脚本使用简单字符串匹配，无法区分：
1. 模型列表中的模型（如 `AI_MODELS.openai`）
2. 默认配置中的默认值（如 `DEFAULT_AI_CONFIG.model`）

**修复建议**：
```bash
# 应改为只统计 AI_MODELS 对象内的模型数量
# 使用 sed 提取 AI_MODELS 对象块，再统计其中的模型数量
```

## 经验教训

1. **验证脚本的局限性**：基于字符串匹配的检查方法无法处理复杂数据结构，会产生误报

2. **默认值与列表的区别**：`DEFAULT_AI_CONFIG.model` 是系统默认值，不应计入模型列表一致性检查

3. **硬编码 vs 导入**：Welcome.tsx 和 Settings.tsx 仍在硬编码模型列表，应重构为从 constants.ts 导入

## 附录：验证脚本误报详解

### constants.ts 中的 gpt- 出现位置
```
第22行:   model: 'gpt-4.1',        ← DEFAULT_AI_CONFIG.model（不应计入）
第47行:   { value: "gpt-5"       ← AI_MODELS.openai[0]
第48行:   { value: "gpt-4.5"     ← AI_MODELS.openai[1]
第49行:   { value: "gpt-4.1"     ← AI_MODELS.openai[2]
```

### Welcome.tsx 中的 gpt- 出现位置
```
第38行:   { value: "gpt-5"       ← TEXT_MODELS.openai[0]
第39行:   { value: "gpt-4.5"     ← TEXT_MODELS.openai[1]
第40行:   { value: "gpt-4.1"     ← TEXT_MODELS.openai[2]
```

### 差异原因
- constants.ts 多出 1 个，因为 `DEFAULT_AI_CONFIG.model` 也包含 gpt-4.1
- 验证脚本无法区分模型列表和默认配置
